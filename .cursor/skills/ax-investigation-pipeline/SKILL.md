# AX Investigation Pipeline

## Назначение

Скилл для исследовательских задач по Dynamics AX без написания кода.  
Результат всегда оформляется в `investigationTask.md` в корне проекта.

## Жесткие правила

1. Не писать код и не вносить изменения в X++/XPO в рамках исследования.
2. Не выдумывать факты: если не найдено в коде/данных, явно писать `не найдено`.
3. При неоднозначности задавать уточняющий вопрос пользователю.
4. Любое утверждение в выводах должно иметь источник: XPO, MCP LightRAG, MCP CUS или ответ пользователя.
5. Перед формулировкой `не найдено` по наименованию AOT-элемента обязательно проверить цепочку: `XPO Label` -> расшифровка в `AOT_cus/*.ald` по всем релевантным файлам меток: `AxMIKru.ald`, `AxGMSru.ald`, `AxKORru.ald`, `axSYSru.ald` (пример: `@MIK12108` -> `Состав дропа` в `AOT_cus/AxMIKru.ald`; `@SYS333411` -> системная строка в `AOT_cus/axSYSru.ald`).
6. Если у элемента в XPO нет явного свойства `Label` (например часть классов/таблиц), писать это явно: `явная Label отсутствует в свойствах объекта`, а не общее `не найдено`.
7. Перед запросом у пользователя AOT-списка обязательно проверить наличие элементов в `parserXPO` (классы/таблицы/методы). Если элементы есть локально, использовать их и не писать, что они отсутствуют.
8. При анализе класса с наследованием (`class X extends Y`) обязательно добавить в исследование родительский класс `Y`: сначала искать в `parserXPO`.
9. Если родителя `Y` нет в `parserXPO`, извлекать его сразу в `parserXPO_Private` из `AOT_cus/PrivateProject_CUS_Layer_Export.xpo` (через MCP CUS/индекс или чтение XPO) в рамках ТОЙ ЖЕ итерации исследования.
10. Запрещено оставлять в результате формулировку вида «нужно дополнительно извлечь родителя», если извлечение уже можно выполнить автоматически в текущем запуске.
11. После извлечения родителя обязательно зафиксировать в `investigationTask.md` минимум 2-3 конкретных метода/поля родителя, влияющих на анализ дочернего класса.

## Входные данные

Обязательные входы:
- Инструкция от пользователя (что исследовать, решить, какие варианты найти).

Опциональные входы:
- Файл XPO (проект/сборка классов, таблиц, форм).

## Стартовый протокол

1. Перед началом анализа очистить `investigationTask.md` (полностью удалить прошлое содержимое) и затем формировать отчет заново только по текущей задаче.
2. Проверить, предоставлен ли XPO.
3. Если XPO не предоставлен, спросить пользователя:
   - `Будет файл XPO для анализа или выполняем исследование только по инструкции?`
4. Если пользователь подтверждает работу без XPO, продолжить только по инструкции + MCP.

## Основной пайплайн

### Шаг 1. Локальный анализ XPO (если есть)

1. Разобрать/прочитать доступные элементы из предоставленного XPO.
2. Сопоставить найденные элементы с задачей пользователя.
3. Сформировать первичную гипотезу решения и список пробелов.
4. Для всех ключевых AOT-элементов из вывода проверить `Label` в XPO и расшифровать через метки в `AOT_cus/AxMIKru.ald`, `AOT_cus/AxGMSru.ald`, `AOT_cus/AxKORru.ald`, `AOT_cus/axSYSru.ald` (и при необходимости другие `*.ald` в той же папке); добавлять в результат пару `системное имя -> человекочитаемое наименование`.
5. Если целевой класс наследуется, отдельно разобрать релевантные методы родителя и отметить влияние родителя на поведение дочернего класса.
6. Если родителя нет локально, сначала извлечь его в `parserXPO_Private`, потом продолжать анализ (не переносить это в раздел «следующие шаги»).

### Шаг 2. Проверка на полноту

Считать решение неполным, если выполняется хотя бы одно:
- не хватает связанного кода/объектов для проверки гипотезы;
- в логике остаются противоречия или неоднозначность;
- не хватает контекста по прошлым решениям/ограничениям.

### Шаг 3. Обращение к LightRAG MCP

Если есть пробелы или сомнения:
1. Выполнить поиск в LightRAG по формулировке задачи и связанным сущностям.
2. Сопоставить найденное с XPO-анализом.
3. Отметить, что именно подтверждено, а что осталось неподтвержденным.

### Шаг 4. Добор из CUS MCP

Если после LightRAG видно, что отсутствуют нужные AOT-элементы:
1. Обратиться к MCP слоя CUS.
2. Найти и извлечь недостающие классы/таблицы/формы/методы.
3. Досопоставить с текущей гипотезой и обновить выводы.

### Шаг 5. Стоп-условие

Если нужный код не найден ни в предоставленном XPO, ни в CUS MCP:
1. Остановить исследование.
2. Запросить у пользователя недостающее списком AOT-элементов.
3. До получения данных не делать предположений о реализации.

## Формат результата (всегда `investigationTask.md`)

Обновляй файл `investigationTask.md` в корне проекта в следующем формате:

```markdown
# Investigation Task

## 1) Промпт для разработки (на основе исследования)
- Краткий контекст задачи.
- Что найдено в коде/данных и как это влияет на реализацию.
- Вариант 1: ...
- Вариант 2: ...
- Вариант 3: ... (если релевантно)

## 2) AOT-элементы для модификации
- Class: ...
- Table: ...
- Form: ...
- Query/View/Enum (если применимо): ...

## 3) Риски и сомнения
- Риск: ...
- Почему риск: ...
- Что нужно для снятия риска: ...

## 4) Нехватка данных (если есть)
- Не найдено: ...
- Нужен от пользователя AOT-список: ...
```

## Требования к качеству результата

1. Для раздела `Промпт для разработки` предлагать 2-3 варианта, если есть достаточные данные.
2. Если данных недостаточно — не натягивать варианты, а фиксировать пробелы в `Нехватка данных`.
3. В `AOT-элементы для модификации` включать только реально обнаруженные элементы.
4. В `Риски и сомнения` отдельно отмечать:
   - функциональные риски;
   - риски неполного покрытия кода;
   - риски из-за отсутствующих артефактов.

## Короткий чеклист перед финализацией

- [ ] XPO учтен или явно зафиксирован режим без XPO.
- [ ] LightRAG использован при пробелах/сомнениях.
- [ ] CUS MCP использован, если не хватает AOT-кода.
- [ ] При отсутствии кода сформирован запрос пользователю списком AOT.
- [ ] `investigationTask.md` обновлен по шаблону.
