# AX Phased Dev Pipeline

## Назначение

Скилл для **реализации** задач Dynamics AX (X++ в `parserXPO`) по постановке, разбитой на фазы.  
Отличается от `ax-investigation-pipeline`: здесь **допускаются** правки `.xpp`, но **строго по одной фазе** и только после явной команды пользователя на старт/следующую фазу.

## Жёсткие правила

1. **Не начинать правки кода**, пока пользователь не дал команду на конкретную фазу (`фаза 0`, `фаза 1`, `продолжай`, `следующий этап`).
2. После **каждой** фазы с изменениями `.xpp` — субагент **`xpp-review`** (readonly); исправить все **critical** и **medium**; повторять review до нуля замечаний.
3. **`/xpo-roundtrip` / `*_WR.xpo`** — **только** по явной команде пользователя (`собери WR`, `/xpo-roundtrip`). Без согласования не запускать `xpo_writer.py`.
4. Перед **любой** правкой `.xpp`: прочитать `.cursor/rules/comment_rules.mdc` и `commentmeta.json` (`developer`, `project`; дата в комментариях — **сегодня**).
5. Код X++ только в `parserXPO/**/*.xpp`; объекты из CUS — через `indexXPO_cus` / `parse_object_from_index` / `/xpo-parse`, не парсить весь CUS целиком.
6. Не выдумывать AOT/API: при пробелах — `/ax-investigation` или `@lightrag ? …`, либо запрос списка AOT у пользователя.
7. Не создавать файлы в репозитории без необходимости фазы; **не** менять `investigationTask.md` (это артефакт investigation-скилла).

## Входные данные

Пользователь передаёт в сообщении с `/ax-phased-dev`:

| Параметр | Обязательность | Пример |
|----------|----------------|--------|
| `PROJECT_ID` | да | `MRC_MERK_000002479_01` |
| `PROJECT_FOLDER` | да | `MRC_MERK_000002479_01NewUPDNumder` |
| Постановка | да | файл **или** текст в блоке `<<< … >>>` |
| `XPO` | нет | `XPO/SharedProject_….xpo` или «извлечь из CUS по индексу» |

**Постановка — один из вариантов:**

- `A)` `@Projects/{PROJECT_FOLDER}/Documentation/task.md`
- `B)` текст в чате:

```text
<<<
{постановка с клавиатуры}
>>>
```

Если нет ни файла, ни блока `<<< >>>` — один короткий вопрос: «Постановка: файл или текст?»

## Стартовый протокол (фаза 0)

Выполнять **только** по команде `фаза 0` / «начни с подготовки»:

1. `/xpo-index-check` (при FAIL → `/xpo-index-cus`, повтор).
2. Прочитать постановку; сверить `commentmeta.json` → `project` = `PROJECT_ID`.
3. Составить **в чате** план фаз (таблица ниже); по запросу пользователя — `Projects/{PROJECT_FOLDER}/Documentation/todoList.md`.
4. Список AOT для извлечения/правок (без правок кода на фазе 0).
5. Краткий отчёт и **стоп** — ждать `фаза 1` или уточнений.

## Фазы (шаблон)

| Фаза | Содержание | Артефакт |
|------|------------|----------|
| **0** | Индекс, постановка, план, scope | отчёт в чате |
| **1** | Разведка: цепочки вызовов, поля, метки (ALD), родители `extends` | карта в чате; при необходимости `/ax-investigation` |
| **2** | Извлечение AOT в `parserXPO` / `parserXPO_Private` | объекты в репо |
| **3…N-2** | Правки по блокам постановки (таблицы, классы, формы — как в задаче) | `.xpp` + xpp-review |
| **N-1** | Проверка без боевого контура (сухой JSON/XML, логи, сценарии) | инструкция теста в чате |
| **N** | Сборка WR | только по команде пользователя |

Нумерацию подзадач (3a, 3b…) допускается уточнять под конкретную постановку на фазе 0.

## Пайплайн одной фазы (1…N-2)

1. Выполнить **только** задачи текущей фазы.
2. Запустить субагент **`xpp-review`** по всем изменённым `parserXPO/**/*.xpp`.
3. Исправить critical/medium; повторить review.
4. Отчёт в чате: сделано, файлы, verdict review, оставшиеся low/риски.
5. **Стоп** — не переходить к следующей фазе без команды пользователя.

## Субагент xpp-review (промпт-шаблон)

Передать субагенту:

- `commentmeta.json` → `project`
- список изменённых `.xpp`
- выдержка постановки (файл или текст пользователя)
- вопрос: critical/medium/low, риски компиляции AX 2012

## Связанные команды и скиллы

| Триггер | Действие |
|---------|----------|
| `/ax-investigation` | только анализ → `investigationTask.md`, **без кода** |
| `/xpo-index-check`, `/xpo-parse`, `/xpo-write`, `/xpo-roundtrip` | инфраструктура XPO |
| `@lightrag ? …` | база знаний проекта |
| `dynamics-ax-xpo-roundtrip` | чеклист XPO↔XPP при правках |

## Формат отчёта после фазы (в чате)

```markdown
## Фаза X — {название}
**Сделано:** …
**Файлы:** …
**xpp-review:** critical 0, medium 0, low …
**Риски:** …
**Дальше:** жду команду `фаза X+1`
```

## Чеклист перед завершением проекта

- [ ] Все фазы постановки закрыты или явно отложены пользователем.
- [ ] Последняя фаза с кодом прошла xpp-review без critical/medium.
- [ ] WR.xpo — только если пользователь явно согласовал сборку.
- [ ] В чате есть сценарии проверки в AX (без выдуманных шагов на проде).
